### **一、开发前期必须完成的操作（关键！）**

#### 1. **获取DeepSeek R1 API权限**

- 必须操作：  
  - 访问 [DeepSeek官网](https://www.deepseek.com/) → 注册开发者账号 → 申请API Key（选择R1模型）。
  - 注意：  
    - 确认API支持**文本分析**（非图片/PDF），R1是文本模型，需确保输入为纯文本（医疗报告需前端预处理为文本）。
    - 申请时勾选`医疗健康`领域权限（部分API需特殊审批）。
    - 密钥安全：  
      - 生成后**绝不硬编码**在代码中！  
      - 通过**环境变量**（如`DEEPSEEK_API_KEY`）或**密钥管理服务**（如AWS Secrets Manager、阿里云KMS）注入。

#### 2. **明确医疗报告输入规范**

- 必须与前端团队对齐：  

  | 输入项       | 说明                                                         | 为什么重要？                                                 |
  | ------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
  | 报告格式     | 前端需提供**纯文本**（非PDF/图片），例如：`"患者：张三，诊断：肺炎，建议：..."` | DeepSeek R1无法解析图片/PDF，前端必须做OCR或文本提取（避免后端重复处理） |
  | 最小文本长度 | 定义阈值（如≥100字符），避免无效请求                         | 防止短文本导致模型输出不完整（如“无异常”）                   |
  | 敏感字段脱敏 | 要求前端在传输前**自动脱敏**（如姓名→“患者1”，身份证号→“110******”） | 符合医疗数据安全法规（如《个人信息保护法》），避免后端处理敏感数据 |

#### 3. **定义质控规则与输出结构**

- 必须输出结构化数据

  （避免模型自由发挥）：  

  ```json
  {
    "score": 85,        // 质控评分（0-100，越高越好）
    "summary": "报告不完整", // 核心问题：过诊/少字/不完整/其他
    "issues": [         // 详细问题列表（数组）
      {"type": "missing", "detail": "缺少影像学检查结果"},
      {"type": "inconsistent", "detail": "诊断与建议矛盾"}
    ],
    "suggestions": [    // 修改建议（数组）
      "补充CT检查结果",
      "修正诊断描述：应为‘细菌性肺炎’"
    ]
  }
  ```

- 为什么关键？

  - 通过**严格Prompt设计**强制模型输出上述JSON（见下文），确保前端能直接解析，避免二次处理。

  - Prompt示例

    （需测试优化）：  

    ```text
    你是一名资深医疗质控专家。请分析以下报告内容，严格按JSON格式输出，禁止额外解释：
    {
      "score": "0-100分，基于完整性、逻辑性、合规性评分",
      "summary": "问题类型（过诊/少字/报告不完整/其他）",
      "issues": [{"type": "问题类型", "detail": "具体描述"}],
      "suggestions": ["具体修改建议"]
    }
    报告内容：{report_text}
    ```

### **二、项目整体框架设计（分层+扩展性优先）**

采用**标准分层架构**，确保未来更换模型/规则无需大改代码：

```plaintext
src/
├── config/                # 配置层（隔离外部依赖）
│   ├── DeepSeekConfig.java  # 读取API Key、Endpoint（从环境变量获取）
│   └── QualityRuleConfig.java # 质控规则配置（未来可外部化为YAML）
├── controller/            # 接口层（只处理HTTP）
│   └── QualityControlController.java # 接收报告文本，返回JSON
├── service/               # 业务逻辑层（核心）
│   ├── QualityService.java  # 主逻辑：调用模型、处理结果
│   └── model/               # 数据模型
│       ├── QualityResult.java # 结构化输出（与前端约定）
│       └── ReportText.java    # 输入校验模型
├── client/                # 外部服务客户端层（隔离模型调用）
│   └── DeepSeekClient.java  # 封装API调用（未来可替换为Qwen/其他模型）
└── util/                  # 工具层
    └── TextSanitizer.java   # 文本脱敏、长度校验（前端已做，此处二次校验）
```

#### **为什么这样设计？**

| 层           | 职责                        | 扩展性体现                                                   |
| ------------ | --------------------------- | ------------------------------------------------------------ |
| **配置层**   | 读取密钥、规则              | 更换模型只需改`DeepSeekConfig`，无需改动业务代码             |
| **客户端层** | 封装API调用                 | 若换用Qwen，只需重写`DeepSeekClient`，`QualityService`完全不变 |
| **业务层**   | 质控逻辑（Prompt+结果处理） | 规则变化通过`QualityRuleConfig`调整，无需修改代码            |
| **接口层**   | HTTP请求/响应               | 与前端交互格式固定，前端修改不影响后端逻辑                   |

### **三、核心执行逻辑（HTTP请求处理流程）**

**关键：所有逻辑在业务层（Service）完成，避免Controller臃肿**

1. **前端请求**：
   `POST /api/v1/quality-control`  

   ```json
   {
     "report_text": "患者：张三，诊断：肺炎，建议：...（纯文本）"
   }
   ```

2. **后端处理流程**（伪代码逻辑）：

   ```plaintext
   1. Controller 接收请求 → 验证 report_text 长度/格式（调用 ReportText.validate()）
   2. Service 层：
        a. 调用 TextSanitizer 脱敏报告文本（二次保险）
        b. 构造 Prompt：拼接 report_text + 固定质控指令
        c. 调用 DeepSeekClient.sendRequest(prompt)
        d. 解析模型返回的 JSON（使用 Jackson 反序列化到 QualityResult）
        e. 附加规则校验（如：若 score < 60，强制添加 "高风险" 标签）
   3. 返回结构化结果给前端（QualityResult）
   ```

3. **关键控制点**：

   - **输入校验**：前端传来的文本必须通过`ReportText`验证（长度、特殊字符过滤）。

   - **模型调用**：`DeepSeekClient`处理重试（如网络失败重试2次）、超时（设置5秒超时）。

   - 错误处理

     ：  

     - 模型返回非JSON → 返回`400 Bad Request` + 错误码（如`MODEL_INVALID_RESPONSE`）  
     - API Key无效 → 返回`401 Unauthorized`  
     - 报告文本无效 → 返回`400 Bad Request`（前端需处理）

### **四、必须注意的事项（避免踩坑）**

#### 1. **安全与合规（医疗项目红线）**

- ✅ 

  必须做：  

  - 所有数据传输用HTTPS（Spring Boot默认支持）。
  - 服务端**不存储**原始报告（仅临时处理，请求结束后清除）。
  - 日志中**脱敏**所有字段（如日志记录`report_text: "患者1, 诊断: 肺炎"`，而非原文）。

- ❌ 

  绝对禁止：  

  - 将敏感数据（姓名/身份证）存入数据库或日志。
  - 使用`System.out.println`打印报告内容。

#### 2. **模型调用的稳定性**

- 关键措施：
  - 设置**API调用超时**（如5秒），避免阻塞线程。
  - 添加**熔断机制**（使用Resilience4j）：若DeepSeek连续失败3次，降级返回默认质控结果（如`score=50, summary="模型服务不可用"`）。
  - **监控**：集成Prometheus，监控API调用成功率、延迟。

#### 3. **未来扩展性设计**

| 扩展场景                     | 如何实现（无需大改代码）                                     |
| ---------------------------- | ------------------------------------------------------------ |
| 更换大模型（如Qwen）         | 1. 新建`QwenClient`类实现`ModelClient`接口 2. 修改`DeepSeekConfig`中的模型类型 → 服务自动切换 |
| 新增质控规则（如“用药冲突”） | 1. 在`QualityRuleConfig`中添加新规则 2. `QualityService`中增加规则校验逻辑（通过策略模式） |
| 支持PDF报告输入              | 1. 前端增加OCR服务（如Tesseract） 2. 后端仅需接收文本，不改动质控逻辑 |

#### 4. **性能与成本考量**

- 避免长文本问题：  
  - DeepSeek R1有上下文长度限制（约32K tokens），需在`TextSanitizer`中截断超长报告（保留关键段落）。
- 成本控制：  
  - 记录每次调用的token数（`DeepSeekClient`中计算），避免无效请求。
  - 对短报告（<500字符）自动跳过模型调用（返回默认结果，如`score=80, summary="内容简短"`）。
